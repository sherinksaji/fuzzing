50.053 Software Testing and Verification

Fuzzing Bluetooth Low Energy (BLE) Zephyr stack.

Similarly to the previous projects, the goal of this task is to implement a fuzzer, but in this task, you will be testing a Bluetooth Low Energy (BLE) Peripheral Application. BLE is a wireless communication protocol designed for low-power, short-range communication. It's commonly used in various IoT devices such as wearables, smart home devices, and healthcare sensors. BLE operates in the 2.4 GHz ISM (Industrial, Scientific, and Medical) band and is known for its low energy consumption, making it ideal for battery-operated devices.

Peripheral Device: A peripheral device in BLE acts as a slave that advertises its presence and responds to connection requests from central devices. Once connected, it communicates data to the central device based on GATT profiles and services. Examples of peripheral devices include sensors, fitness trackers, and smart home devices.

Central Device: A central device in BLE acts as a master that initiates connections and controls data exchange with peripheral devices. It scans for advertising peripheral devices, establishes connections, and manages data transactions. Examples of central devices include smartphones, tablets, and computers.

 

Figure 1. Bluetooth Low Energy protocol stack.

BLE Stack: The Bluetooth low energy protocol stack (or protocol stack) consists of the controller and the host. This separation of controller and host derives from the implementation of classic Bluetooth BR/EDR devices, where the two sections are implemented separately. Any profiles and applications sit on top of the GAP and GATT layers of the protocol stack

Controller Layer: The controller layer is responsible for managing the physical layer operations, including radio communication, modulation, and signal processing. It handles tasks such as frequency hopping, packet encoding/decoding, and error checking.

  • Physical Layer (PHY): BLE operates in the ISM band from 2.4 to 2.5 GHz, and defines 40 channels. Three channels (37, 38 and 39) are dedicated to the advertising mode (allowing devices to broadcast data using some packets named advertisements), while the 37 others channels (numbered from 0 to 36) are dedicated to the connected mode, which is used when a connection is established between two devices.

Figure 2. Mapping between the frequencies and Bluetooth LE channels.

  • Link Layer (LL): The Link layer is responsible for scanning, advertising, and creating/maintaining connections and directly interfaces to the physical layer (PHY).

  • Host Controller Interface (HCI): It provides communication between the host and controller through a standardized interface. This layer can be implemented either through a software API or by a hardware interface such as universal asynchronous receiver / transmitter (UART), serial peripheral interface (SPI), or USB. The Specification of the Bluetooth System describes Standard HCI commands and events.

Host Layer:

The host layer manages higher-level protocol operations and interactions with applications. It includes protocols like the Attribute Protocol (ATT), Generic Attribute Profile (GATT), and Security Manager (SM). The host layer handles data exchange, connection management, and security procedures.

  • Logical Link Control and Adaptation Protocol (L2CAP): The L2CAP sits on the top of HCI layers and enables the data transfers between the upper layers of the host (GAP, GATT, application) and the lower layer BLE protocol stack.

  • Attribute Protocol (ATT): It defines how data/attributes are represented in a BLE server database and also defines methods by which that data/attribute can be read or written.

  • Security Manager (SMP): the Security Manager layer defines the methods for pairing and key distribution, and provides functions for the other layers of the protocol stack to securely connect and exchange data with another device. The Security Manager provides a set of pairing and bonding procedures to negotiate multiple keys dedicated to increase the security level of the connection. One of the most important keys is the Long Term Key, which allows to establish an AES-CCM encryption over the Link Layer to avoid eavesdropping..

  • Generic Access Profile (GAP): GAP helps to determine how the BLE devices can or can not interact with each other and is responsible for connection functionality.

  • Generic Attribute Profile (GATT): GATT defines the way two Bluetooth Low Energy (BLE) devices transfer data back stored by Attribute protocol (ATT).

Application Layer: The application layer is the uppermost layer like any other type of system. It is responsible for the user interface, containing logic, and data handling for everything related to the actual use-case that the application implements. It includes profiles and services defined by the Bluetooth SIG (Special Interest Group) for various applications (e,g,. heart rate monitoring, proximity sensing, and data transfer).

In summary, the BLE stack consists of controller, host, and application layers, with central and peripheral devices playing distinct roles in communication. The central device initiates connections and controls data exchange, while the peripheral device responds to connection requests and communicates data based on predefined profiles and service (see the Zephyr BLE stack reference).

Messages exchanged during a BLE connection

Initially, the peripheral (i,e,. smart watch) periodically broadcasts advertisements to nearby devices and the central starts in the scanning state. The central scans for such advertisements and gets further information from the peripheral such as its name by sending a scan request (i,e., $1$ in Figure 2).

Figure 3. BLE messages exchange during a connection [2].

Relevant BLE procedures: Although BLE involves many messages exchanged over-the-air between two devices, this project aims into exploring the implementation security of the BLE Application layer. In this context, few BLE procedures need to be understood to properly design a fuzzing tool targeted to BLE APPs. Such procedures are listed below:

  • Scan Request/Scan Response:

    • Scan Request (1) : In this step, the central device (e.g., smartphone) sends a scan request to discover nearby advertising peripherals.

    • Advertisement: The peripheral device (e.g., BLE-enabled sensor) responds with an advertising packet containing information about itself.

    • Scan Response (2): Optionally, the peripheral device may send a scan response packet with additional information.

  • Connection Request/Connection Response:

    • Connection Request (3): The central device sends a connection request to the peripheral device it wishes to connect to.

    • Connection Response: The peripheral device responds with a connection response, indicating whether the connection is accepted or rejected.

  • ATT Request/Response or GATT Services Discovery: After the connection is established and secured, the central device may initiate Attribute Protocol (ATT) requests to discover services and characteristics offered by the peripheral device. The peripheral device responds with Attribute Protocol (ATT) responses containing information about its services, characteristics, and descriptors.

    • Services: Services in Bluetooth Low Energy (BLE) represent collections of related functionality offered by a peripheral device. They organize and encapsulate specific features or capabilities provided by the device. Each service is identified by a unique 16-bit or 128-bit UUID (Universally Unique Identifier), allowing the central device to identify and interact with different functionalities of the peripheral.

    • Characteristics: Characteristics are attributes within services that hold data or provide features. They encapsulate specific data points or functions and are identified by a 16-bit or 128-bit UUID within their respective service. Characteristics can be read, written, or subscribed to for notifications or indications, allowing for bidirectional communication between the central and peripheral devices.

    • Descriptors: Descriptors further define characteristics, specifying additional properties or behaviors. They provide metadata about a characteristic's value, such as its format, unit, or client configuration. Descriptors are identified by a 16-bit UUID and are nested within their corresponding characteristics. They enhance the capabilities and understanding of characteristics, enabling more precise interaction between devices during data exchange.

 

BLE Zephyr Stack Simulation Setup (e.g., BLE Device Emulation)

The Zephyr Project is an open-source, real-time operating system (RTOS) designed for resource-constrained embedded systems. It supports various architectures and platforms, including those with Bluetooth Low Energy (BLE) capabilities.

When it comes to BLE implementation in the Zephyr stack, it provides a comprehensive set of APIs and protocols for developing BLE-enabled applications. Install/Download dependencies

We provide the binary of the Zephyr APP already compiled (zephyr.exe). This binary can directly run on Linux systems such as Ubuntu 22.04. However, since the binary is compilated to simulate 32bit systems (e.g., microcontrollers), we need to install the gcc multilib system package, and other coverage tools such as gcov and lcov.

xxxxxxxxxx
sudo apt install gcc-multilib gcovr lcov -y

Then, extract the compressed folder from this project and install its python requirements:

xxxxxxxxxx
tar -xf ble-fuzzing-project.tar.gz # Extract package
python3 -m pip install "./bumble" # Install bumble ble stack library
chmod +x zephyr.exe # Make target binary executable

Running the Target (Zephyr stack)

To run the target using the simulation setup, first we need to start the software which "simulates" a BLE Controller hardware (run_ble_tester.py). This is done so we can test the Zephyr Stack (target) without requiring the use of a BLE dongle.

Open two terminals. In the first terminal, run the following command to start the software controller:

xxxxxxxxxx
python3 run_ble_tester.py tcp-server:127.0.0.1:9000

The output should be the following:

xxxxxxxxxx
>>> Waiting connection to HCI...
>>> Connected
Waiting Advertisment from BLE Target

In the second terminal, run the following command to run the target binary:

xxxxxxxxxx
GCOV_PREFIX=$(pwd) GCOV_PREFIX_STRIP=3 ./zephyr.exe --bt-dev=127.0.0.1:9000

The output should be the following:

xxxxxxxxxx
*** Booting Zephyr OS build v3.6.0-rc2-130-g29d7f9fa73da ***
[00:00:00.000,000] <inf> fs_nvs: nvs_mount: 4 Sectors of 4096 bytes
[00:00:00.000,000] <inf> fs_nvs: nvs_mount: alloc wra: 0, fd0
[00:00:00.000,000] <inf> fs_nvs: nvs_mount: data wra: 0, 19

The previous command started the zephyr stack and connects it to a HCI controller that is at address 127.0.0.1:9000. The output in terminal 1 should indicate that a device has been discovered through receiving BLE advertisements:

Note that all services have some characteristic listed and informs whether we can read or write to them. For example, service "Generic Access" has a characteristic named "Device Name", which we can only read. Moreover, the "handle" informed in all attributes is considered as the address where an information is located in the target. Every service, characteristic and descriptor exposed by the target has a handle address that you can read from or write to.

 

Sending inputs to the BLE Target:

After the script run_ble_tester.py connects to the BLE target, it will try to read and write to every service or characteristic of it and inform whether the operation was successful. If any read or write operation was fails, an error message indicating the cause is shown:

Consequently, run_ble_tester.py can be used to design a fuzzing driver since this code allows us to send arbitrary ATT read or write requests to the target.

For example, you can modify the function on_connection to add your fuzzing algorithms. Currently, this function reads all attributes from the target and only writes byte 0x01 to each attribute:

xxxxxxxxxx
        # -------- Main interaction with the target here --------
        print('=== Read/Write Attributes (Handles)')
        for attribute in attributes:
            await write_target(target, attribute, [0x01])
            await read_target(target, attribute)

Calculating the coverage

We have provided a build.tar.gz folder that contain the relevant *.gcno files when building the application that you previously uncompressed. This files are neccesary in case you want to calculate the coverage (i.e,. using gcov). Additionally, when running the application, the respective .gcda files are created withing the same /build directory. An overview of the gcov flow is better depicted in Figure 5.

Figure 5. General gcov flow.

You can use lcov command to generate code coverage information into a file lcov.info:

xxxxxxxxxx
lcov --capture --directory ./ --output-file lcov.info -q --rc lcov_branch_coverage=1

This will create a file with the coverage information, you can interpret the abbreviations of this file are as follows:

xxxxxxxxxx
TN: test name
SF: source file path
FN: function name
FNF:  number functions found
FNH: number hit
BRDA: branch data: line, block, (expressions,count)+
BRF: branches found
DA: line number, hit count
LF: lines found
LH:  lines hit.

For instance, we can create a short summary of the coverage rate as follows:

xxxxxxxxxx
lcov --rc lcov_branch_coverage=1 --summary lcov.info

A sample output is shown below:

xxxxxxxxxx
Reading tracefile lcov.info
Summary coverage rate:
  lines......: 35.4% (6474 of 18302 lines)
  functions..: 45.3% (991 of 2187 functions)
  branches...: 18.8% (2010 of 10712 branches)

Finally, you can generate a html report using:

xxxxxxxxxx
genhtml lcov.info --output-directory lcov_html -q --ignore-errors source --branch-coverage --highlight --legend
